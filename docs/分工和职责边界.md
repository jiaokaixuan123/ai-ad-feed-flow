## 横跨部分的职责划分

这五块是两人必然交叉的地方，每块明确**主导方 / 配合方 / 交界物**：

**1. 数据模型与接口契约 — `data/model/*` + 后端 JSON**
主导：乙。配合：甲提渲染需求（卡片要展示哪些字段）。交界物：一份写进 `docs/` 的契约表。规则：模型字段的增删改由乙落地，但任何改动改前先同步。MVP 的 `AdType`（三种）要先和设计文档（四种）对齐。

**2. UI 状态契约 — `FeedUiState` / `DetailUiState`**
主导：乙实现，因为它由 Repository 和 StateFlow 驱动。配合：甲定义"要哪些状态字段"（如三态需要 `error: String?`、空态需要 `isEmpty`）。交界物：`FeedUiState` 的字段定义。甲只 collect 这个 state，绝不自己造状态变量。

**3. 图片加载 — Glide**
拆成两半：乙封一个统一加载入口（含占位图、失败重试、缓存配置），甲在 ViewHolder 里调这个入口（`load(coverUrl)`）。交界物：那个加载入口的方法签名。不允许甲在 cell 里散写 Glide 配置。

**4. 播放器 — ExoPlayer 池 + cell 内渲染（最易翻车，结对做）**
拆成两半：乙负责播放器池的生命周期（创建 1~2 个实例、attach/detach 绑定解绑、release、SimpleCache）；甲负责 `ExoPlayerView` 在 `VideoAdViewHolder` 里的摆放、滑出屏幕暂停的时机（和滚动事件耦合）。交界物：`bindPlayer(holder)` / `unbindPlayer(holder)` 这一对方法。这块设计文档明确要求结对、早做，别一个人扛。

**5. Mock / 网络切换开关 — `AppGraph`**
主导：乙（在 `AppGraph` 里加一个 flag，一行切 `MockFeedDataSource` ↔ `RetrofitFeedDataSource`）。意义：这是两人不互相阻塞的命门——乙做后端期间，甲始终用 Mock 跑 UI，后端没好也不卡甲。这是 MVP 现有抽象留下的最大红利。

一句话守住边界：**甲这侧的代码里不应该出现 Retrofit、Glide 配置、StateFlow 的创建；乙这侧的代码里不应该出现 XML、ViewHolder、动画。** 谁的代码里冒出了对方领域的东西，就是边界被破坏的信号。
