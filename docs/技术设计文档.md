1. # 单列广告信息流 App — 技术设计文档

   > 客户端工程训练营实战作业 · Android · 2 人小组 · 周期 5/26–6/7 团队此前为后端背景、客户端零基础。本设计的核心策略：**用后端能力托管「数据 + AI」，让客户端专注现学的「列表 / 播放器 / 交互」。**

   ------

   ## 1. 项目概述

   实现一个内容与广告混排的**单列信息流 App**：客户端展示单列信息流、多样式广告卡片与详情页交互；后端提供分页数据并通过云端大模型为每条广告预生成摘要与智能标签，进一步支持标签过滤与对话式搜索。

   **范围划分（一句话记住）：**

   - **后端负责**：内容数据（广告、摘要、标签）+ AI（预生成、对话搜索）。
   - **客户端负责**：列表与播放器、点赞/收藏/分享、曝光/点击统计、标签过滤、所有 UI 与动画。
   - **媒体素材**（图片/视频）走远程 URL，由客户端本地缓存。

   **约束**：2 周、2 人、零客户端经验 → 先把核心功能跑稳，AI 与动效为加分项，必要时按第 9 节降级。

   ## 2. 技术选型

   | 领域               | 选型                                           | 理由                                                         | 对应需求               |
   | ------------------ | ---------------------------------------------- | ------------------------------------------------------------ | ---------------------- |
   | 语言               | **Kotlin**                                     | 官方主流、协程处理异步、样板代码少                           | 全局                   |
   | UI 体系            | **View 体系 + XML + ViewBinding**              | 题目核心词（cell 复用 / 播放器复用 / 缓存池）是命令式 View 体系概念，答辩可讲、范例多；Compose 自动复用反而无从展示 | 核心交互、资源复用     |
   | 列表               | **RecyclerView + ListAdapter + DiffUtil**      | 多样式卡片用 `getItemViewType`；DiffUtil 做局部刷新          | 单列信息流、多样式卡片 |
   | 频道 Tab           | **TabLayout + ViewPager2**                     | 每个频道一个 Fragment，各持 RecyclerView + ViewModel         | 顶部 Tab 切换          |
   | 页面跳转           | **多 Activity**（Feed → Detail）               | 对零基础最直观；跨页状态靠单例 Repository 同步               | 详情页、状态同步       |
   | 架构               | **MVVM**（ViewModel + StateFlow + Repository） | 职责清晰、UI 自动订阅刷新                                    | 数据/状态管理          |
   | 图片               | **Glide**                                      | 自带内存 + 磁盘缓存、占位图、失败重试                        | 图片加载与缓存         |
   | 视频               | **Media3 ExoPlayer + SimpleCache**             | 列表内播放经典方案、范例多；SimpleCache 做磁盘缓存           | 视频卡片、播放器复用   |
   | 网络               | **Retrofit + OkHttp + Moshi**                  | 标准网络层，便于做超时/重试/错误码/Mock 切换                 | 数据获取、网络层封装   |
   | 本地持久化（可选） | **Room / DataStore**                           | 让点赞、统计重启后仍在（加分项，非必须）                     | 状态管理、统计         |
   | 统计可视化         | **MPAndroidChart**                             | 快速出柱状/饼图展示曝光/点击                                 | 统计可视化             |
   | 后端               | **FastAPI（Python）或 Spring Boot**            | 发挥团队后端长处；一个服务同时托管 feed + AI                 | 数据层、AI 接口        |
   | AI 模型            | **通义千问（阿里云百炼，OpenAI 兼容接口）**    | 新用户每模型约 100 万 token / 90 天免费，足够；摘要标签预生成只需调一次 | AI 摘要、标签、搜索    |
   | AI 结果存储        | **JSON 文件 / SQLite**                         | 30~60 条广告规模，**不用 redis**（过度设计）                 | AI 缓存策略            |

   ## 3. 整体架构

   ```
   ┌─────────────────────────── Android 客户端 ───────────────────────────┐
   │  UI 层 (Activity / Fragment / RecyclerView / ExoPlayerView)          │
   │        │ 订阅 StateFlow / LiveData                                   │
   │  ViewModel 层 (每个信息流页一个，持有 UI 状态，扛配置变化)          │
   │        │                                                            │
   │  Repository 层 (App 级单例 = 单一数据源)                            │
   │    ├── FeedRepository      ← 网络：分页内容 + 摘要 + 标签            │
   │    └── InteractionStore    ← 本地：点赞/收藏/分享 + 曝光/点击事件    │
   │        │                                                            │
   │  数据源：Retrofit 网络 / 本地 mock(降级) ; Glide / ExoPlayer 缓存   │
   └──────────────────────────────────┬───────────────────────────────┘
                                       │ HTTP (JSON)
   ┌──────────────────────────────────┴───────────────────────────────┐
   │  轻量后端 (FastAPI / Spring Boot)                                  │
   │    GET  /feed?channel=&page=     分页广告（内嵌已生成的摘要/标签）  │
   │    POST /search                  对话式搜索 → 返回匹配广告 id        │
   │    [离线脚本] 批量调 Qwen 预生成摘要/标签 → 校验 JSON → 存缓存      │
   │  媒体素材：远程图片/视频 URL（picsum / 公开 sample mp4）           │
   └────────────────────────────────────────────────────────────────────┘
   ```

   **两类数据严格分开，渲染时才合并**：UI 看到的卡片 = 内容数据（网络）+ 叠加本地交互状态（"这条 id 点过赞没"）。这套设计让信息流与详情页天然同步，也是答辩重点。

   ## 4. 功能模块与解决思路

   ### 4.1 单列信息流 + 多样式卡片（核心）

   RecyclerView 用 `getItemViewType()` 按数据类型返回大图 / 小图 / 图文 / 视频四种类型，对应四个 ViewHolder。ListAdapter + DiffUtil 负责差量更新，避免整表刷新。

   ### 4.2 频道 Tab 切换

   TabLayout + ViewPager2，每个频道一个 Fragment，各自持有 RecyclerView 与 ViewModel。题目要求**切换时刷新数据**：切到某频道时按 `channel` 拉取该频道数据。多个 RecyclerView 共享一个 `RecycledViewPool`，跨频道复用 ViewHolder（答辩亮点）。

   ### 4.3 详情页 + 返回保持位置

   点击卡片携带广告 id 跳转 DetailActivity。返回时只要不重建 adapter，RecyclerView 自动保持滚动位置；Tab 间切换则让 Fragment 存活或保存 LayoutManager 状态。

   ### 4.4 下拉刷新 / 上拉加载

   SwipeRefreshLayout 做下拉刷新（清缓存重拉第 1 页）；RecyclerView 滚动监听到接近底部时加载下一页并追加。需处理**加载中 / 空态 / 错误态**三种状态。

   ### 4.5 资源复用（核心，答辩重点）

   - **Cell 复用**：RecyclerView ViewHolder 回收机制 + 跨频道共享 RecycledViewPool。
   - **播放器复用**：维护一个极小的 ExoPlayer 池（1~2 个）。cell attach 时绑定播放器、detach 时解绑回收；滑出屏幕暂停、销毁时 release。**列表内放播放器是全项目最难、最出彩的一块，需早做、结对攻坚。**
   - **缓存池**：图片由 Glide（内存+磁盘）自动管理；视频由 ExoPlayer SimpleCache（磁盘）管理。

   ### 4.6 数据 / 状态同步

   单一数据源（InteractionStore 单例）持有 `Map<广告id, 交互状态>` 的 StateFlow。信息流与详情页都订阅它，所以**详情页点赞，返回信息流自动同步**。

   ### 4.7 视频播放器控制

   外流视频默认暂停，点击播放；进入详情页（内流）自动播放；支持暂停/续播、静音/音量。播放器实例从 4.5 的池中取。

   ### 4.8 动画

   页面切换用 Activity transition；点赞做"彩蛋"动效（缩放 + 粒子/心形动画）。

   ### 4.9 埋点统计（本地模拟）

   - **点赞/收藏/分享**：纯本地，写入 InteractionStore。
   - **曝光口径（务必写清）**：cell 可见面积 **≥50% 且持续 ≥500ms** 记一次曝光，同一次出现去重；点击在 onClick 记录。事件存本地列表（可选落 Room）。
   - **可视化**：用 MPAndroidChart 展示各广告曝光/点击次数。

   ### 4.10 AI 摘要与标签（可选加分）

   后端**离线一次性**把每条广告喂给 Qwen，要求严格 JSON 输出（摘要 + 结构化标签：品类/风格/受众/场景），做 **schema 校验，格式不对则重试**，通过后存缓存，随 feed 接口下发。卡片用摘要做副标题、标签做 chips。生成失败降级回标题。

   ### 4.11 标签点击过滤（可选加分）

   标签已随 feed 下发并加载进内存，点 chip 即在已加载列表上本地过滤 + DiffUtil 刷新，**无需后端**。边界：分页下本地过滤只覆盖已加载项；本项目数据量小，采用"一次性多加载 + 本地过滤"，答辩可补充"扩展时把过滤下推到后端查询"。

   ### 4.12 对话式搜索（可选加分）

   目录小，**不用向量库**：把全部广告 (id, 标题, 标签, 摘要) 拼进 prompt，让 Qwen 返回排序后的匹配 id + 理由（JSON），客户端按 id 渲染结果。聊天式 UI。答辩"如何扩展到百万级"答：embedding + 向量检索。

   ### 4.13 图片 / 视频加载与缓存（可选加分）

   Glide：占位图 + 失败重试 + 首屏优先；ExoPlayer SimpleCache 缓存视频片段。均为客户端本地，不涉及后端 DB/redis。

   ## 5. 数据层设计（生命周期与缓存）

   **按来源分两类**：内容数据（网络）/ 交互状态（本地），互不混淆，渲染时合并。

   **按作用域分层**：

   - ViewModel：UI 状态，Fragment/Activity 级，扛配置变化。
   - Repository：App 级单例，活得比页面长 → 跨页同步的根基。
   - StateFlow/LiveData：UI 订阅，数据变则 UI 自动刷新。

   **缓存分层（各管各的）**：

   - 网络响应：Repository 内存中按 channel 缓存已加载页；可选落 Room 实现重启秒开；OkHttp 也可开 HTTP 缓存。
   - 图片：Glide 内存 + 磁盘（自动）。
   - 视频：ExoPlayer SimpleCache（磁盘）。
   - AI 结果：后端已缓存并随 feed 下发，客户端不再单独缓存。

   **数据的一生**：进信息流 → ViewModel 向 Repository 要第 1 页 → Repository 先查内存缓存，无则请求网络、缓存、返回 → 滑到底加载下一页追加 → 下拉刷新清缓存重拉 → 切 Tab 按频道重新拉取（各频道缓存与滚动位置分开存）→ 进程存活期间内存缓存有效，落 Room 则重启仍在。

   ## 6. 后端接口设计（约定）

   ```
   GET /feed?channel={featured|ecom|local}&page={n}&size={20}
   → 200
   {
     "page": 1, "hasMore": true,
     "items": [
       {
         "id": "ad_001",
         "type": "big_image | small_image | image_text | video",
         "title": "……",
         "summary": "AI 生成摘要（已缓存）",
         "tags": ["运动", "学生党", "性价比"],
         "cover": "https://.../cover.jpg",
         "videoUrl": "https://.../sample.mp4",   // type=video 时存在
         "images": ["https://.../1.jpg"]          // 图文时存在
       }
     ]
   }
   
   POST /search   body: { "query": "适合学生的平价运动鞋" }
   → 200 { "results": [ { "id": "ad_007", "reason": "平价+运动+学生" } ] }
   ```

   客户端用统一数据模型对接；网络层支持超时/重试/错误码，并保留一个 **本地 mock ↔ 网络** 的切换开关（降级用，也是设计点）。

   ## 9. 风险与降级方案

   **砍功能优先级（落后时从下往上砍）**： 核心信息流 ＞ 播放器复用 ＞ AI 摘要标签（预生成） ＞ 对话搜索 ＞ 花哨动效。

   - 先把 MVP（单 Tab、能滚、进详情、返回保位置）跑通，再加任何 AI/动画。
   - 后端部署遇阻 → 切本地 assets mock（保留 Repository 抽象，图片视频仍走网络）。
   - AI 生成失败 → 摘要降级回标题、标签降级为空。
   - 视频复用搞不定 → 先做"列表静态封面 + 详情页才播放"的简化版。

   ## 11. Git 协作规范（GitHub）

   团队 2 人、周期 2 周，采用轻量的 **GitHub Flow**：`main` 永远可运行，所有改动走 feature 分支 + Pull Request，**不直接 push 到 `main`**。无需 GitFlow 的 develop/release/hotfix 分支（对本项目是过度设计）。

   ### 11.1 分支模型

   - `main`：受保护主干，始终可编译、可运行，只能通过 PR 合入。
   - feature 分支：每个功能/任务从 `main` 切出，完成后 PR 合回，合并后删除。

   分支命名 `<类型>/<简短描述>`，小写中划线：

   | 前缀        | 用途      | 示例                                    |
   | ----------- | --------- | --------------------------------------- |
   | `feat/`     | 新功能    | `feat/feed-list`、`feat/exoplayer-pool` |
   | `fix/`      | 修 Bug    | `fix/video-crash-on-scroll`             |
   | `refactor/` | 重构      | `refactor/repository-cache`             |
   | `docs/`     | 文档      | `docs/api-protocol`                     |
   | `chore/`    | 构建/杂项 | `chore/gitignore`                       |

   ### 11.2 提交信息规范（Conventional Commits）

   格式 `<类型>: <简短描述>`，一句话讲清这次提交干了什么；**小步、频繁提交，一个提交只做一件事**。

   | 类型       | 含义                   |
   | ---------- | ---------------------- |
   | `feat`     | 新功能                 |
   | `fix`      | 修复 Bug               |
   | `refactor` | 重构（不改外部行为）   |
   | `style`    | 格式调整（不影响逻辑） |
   | `docs`     | 文档                   |
   | `test`     | 测试                   |
   | `chore`    | 构建、依赖、杂项       |

   示例：

   ```
   feat: 信息流支持视频卡片自动播放
   fix: 修复快速滑动时播放器未释放导致的内存泄漏
   docs: 补充 feed 接口字段定义
   ```

   ### 11.3 日常协作流程

   ```bash
   # 1. 开工前同步最新 main
   git checkout main
   git pull origin main
   
   # 2. 从 main 切出功能分支
   git checkout -b feat/feed-list
   
   # 3. 开发 + 小步提交
   git add .
   git commit -m "feat: 实现单列 RecyclerView 多样式 cell"
   
   # 4. 推到远程
   git push -u origin feat/feed-list
   
   # 5. 在 GitHub 开 PR（base: main），对方 Review 后合并（建议 Squash and merge）
   
   # 6. 合并后清理
   git checkout main && git pull origin main
   git branch -d feat/feed-list
   ```

   2 人也要走 PR + Review：哪怕对方只是快速扫一眼，既能互相了解进度、发现问题，也给协作留痕（评分看协作）。

   ### 11.4 冲突与同步

   - **每天至少同步一次 `main` 到自己分支**（题目要求每日小队同步）：在功能分支上 `git merge origin/main`（熟练后可用 `git rebase`），尽早解决冲突，别攒到最后。
   - 分工已天然降低冲突：甲改 UI、乙改数据/AI，文件基本不重叠。
   - **共享文件**（数据模型、接口协议、`build.gradle`）改动前先在群里同步一句，避免双方同时改同一处。

   ### 11.5 .gitignore 与密钥安全（重要）

   新建仓库时选 GitHub 的 **Android** `.gitignore` 模板，确保忽略：

   ```
   *.iml
   .gradle/
   /local.properties
   /.idea/
   /build/
   /app/build/
   .DS_Store
   captures/
   ```

   - **API Key 绝不入库**：通义千问 key 放 `local.properties`（已被忽略），经 `BuildConfig` 读取；后端 key 放环境变量。一旦误提交，立即到阿里云控制台**吊销并换新 key**（删一次提交不够，历史里仍能翻出来）。

   ## 附：答辩高频追问准备

   1. **cell 复用 + 多样式**：`getItemViewType` + 多 ViewHolder + DiffUtil 局部刷新。
   2. **列表位置保持**：不重建 adapter 自动保持；Tab 切换保存 LayoutManager 状态。
   3. **播放器复用（最出彩）**：小型 ExoPlayer 池，attach/detach 绑定解绑，滑出暂停，销毁 release。
   4. **跨页状态同步**：单一数据源 = InteractionStore 单例，两页订阅同一份。
   5. **曝光口径**：可见 ≥50% 且 ≥500ms 记一次、去重；说明为什么这样定。
   6. **AI 结构化约束 + 缓存**：JSON schema 校验+重试、按 id 幂等缓存、失败降级。
   7. **图片/视频加载**：Glide 占位+重试+首屏优先；ExoPlayer SimpleCache。
